1. Feature Overview
1.1. What’s news
Currently, our MApp users can visit the booking form without logging in. Without logging in, we observe several problems leading to churned users or longer booking creation due to the hassleness in manually filling in booking contact and passenger details.
After creating a non-login booking, there are several limitations on accessing certain payment methods and managing the booking (refund/reschedule). This results in a higher operational contact rate due to non-login hassleness and less effective retention strategy as users can neither receive Traveloka Points nor upgrade their Priority tier. 

1.2. Impact Analysis
  1. Entry Points
    1. OBF
    2. BFF
    3. Other Flight contact details
  2. User login state
    1. Logged IN
    2. Logged Out
  3. API Error
    1. 4xx status code
      1. Captcha required
      2. Bad Request
    2. 5xx status code
  4. API Encryption
    1. Api encryption applies to all new apis
  5. Regression Potential
    1. The flight team needs to run regression tests on the OBF and BFF booking forms to confirm that the current booking flow works as well.
  6. Risk
    1. Currently, our system sends a verification code directly to the user's email after they fill out the email field in the contact details. There is a potential risk that a fraudster could repeatedly submit non-SSO emails, causing our backend to send OTPs as often as the email is entered. This could impact system stability and increase email-sending costs (likely very small).`
